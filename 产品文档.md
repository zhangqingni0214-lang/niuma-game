# 第一部分 · 立项陈述

## 1.1 项目概述

| 项目名 | 牛马公司 |
|---|---|
| 类型 | 文本驱动 · 黑色幽默 · 轮回养成 · Web 小游戏 |
| 平台 | Web（PC / 移动浏览器），后续考虑微信小游戏 |
| 单局时长 | 5–8 分钟（标准 14 天 = 42 时间槽） |
| 项目阶段 | 原型成熟期，整体完成度约 75–80% |
| 技术栈 | 原生 HTML + CSS + JS，纯静态部署，无后端依赖，localStorage 存档 |
| 美术风格 | 像素风（凤凰点阵体 Vonwaon），暗影 + 钉子装饰，菜单按角色切换背景 |

**一句话定位**：让玩家在 5 分钟里轮回一次"打工人 14 天生存模拟"，用黑色幽默把现实里咽下去的那口气吐出来。

## 1.2 立项背景与市场机会

**情绪供给侧的真空带**：
- 主流职场题材内容（影视、综艺）以"逆袭爽文"或"温情和解"为主，缺少"承认就是烂，但我们一起苦中作乐"的中间层；
- 短视频赛道的"打工人段子"消费旺盛，但停留在被动观看，缺乏**互动消费**载体；
- Web 小游戏赛道（《人生重开模拟器》《我把女朋友卖了 100 万》等）证明了"文本驱动 + 轮回结构 + 段子密度"的可行性，但同主题在"职场细分"上仍有空位。

**目标切口**：把"人生重开"的轮回结构，垂直到职场细分，配以高密度的"嘴硬牛马"段子和有数值依据的死亡多结局，做一款**可分享、可重复、可上瘾**的小品级网页游戏。

## 1.3 目标用户

| 维度 | 主要画像 |
|---|---|
| 年龄 | 18–35 岁 |
| 身份 | 在职白领、互联网从业者、广义"打工人"，含部分校招实习生 |
| 城市 | 一二线为主 |
| 设备 | 移动浏览器 > 桌面浏览器 |
| 触媒习惯 | 小红书 / 微博 / 即刻 / 朋友圈，对"段子梗图"敏感，倾向碎片化消费 |
| 付费意愿 | 极低（游戏本体免费） |
| 分享意愿 | 高（结局截图、技能名、段子标签都具备社交资产属性） |

## 1.4 核心体验主张（Pillars）

1. **5 分钟苦中作乐**：一局短，节奏快，碎片化也能完整体验。
2. **段子即玩法**：选项文本本身就是笑点，不仅是数值的载体。
3. **死得越离谱越好笑**：13 种结局都是高密度黑色幽默，活下来反而是惊喜。
4. **轮回有积累**：业力—技能—新职业—新结局，让"再来一次"有动力。
5. **嘴硬即正义**：怼老板的选项始终存在，且角色性格会让"嘴硬"变成有数值价值的策略。

## 1.5 竞品坐标

| 竞品 | 优势 | 我们差异 |
|---|---|---|
| 人生重开模拟器 | 轮回结构成熟，话题度高 | 我们更聚焦"职场"垂类，互动密度更高（每槽都是 2–4 选项的事件），技能树提供周目沉淀 |
| 中国式家长 | 玩法成熟，养成深 | 我们更轻量，单局 5 分钟而非 1 小时，移动友好 |
| 抖音 / 小红书职场段子 | 段子产能高，触达广 | 我们提供"互动消费 + 结局生成"，从被动看变成主动玩 |

## 1.6 商业模式与发行策略

**当前版本**：免费 Web 游戏，纯静态部署成本接近零，发行重心在病毒传播。

**潜在变现路径**（按优先级排序）：
1. **结局分享卡片**——结局自动生成可分享图（人设标签 + 段子）→ 流量入口；
2. **皮肤 / 角色扩展**——加入"程序员""设计师""产品经理"等垂直职业，做轻量 IP 联名；
3. **微信小游戏化**——接入小游戏广告分账；
4. **企业向定制版**——B 端公司年会 / Team Building 定制版本（"我们公司的牛马"）。

**短期不考虑**：直接付费、订阅、Gacha。会破坏"自嘲对话"的体验调性。

## 1.7 技术架构与团队配置

**技术栈**：纯前端单页（index.html + 5 个 JS 文件 + 1 个 CSS），无构建依赖，可直接静态部署到任意 CDN / GitHub Pages / 微信内嵌 H5。

| 文件 | 行数 | 职责 |
|---|---|---|
| game.js | 1099 | 核心循环、状态、UI 渲染、Toast / Modal、存档 |
| events.js | 1476 | 49 个事件数据 |
| jobs.js | 123 | 2 角色 + 7 职业 |
| skills.js | 130 | 10 技能定义 + 前置判定 |
| endings.js | 136 | 13 结局 + 15 人设标签 |
| index.html | 211 | 6 个屏幕的 DOM 结构 |
| styles.css | 1194 | 像素风样式 + 动效 |

**团队最小配置**：1 策划 + 1 程序 + 0.5 美术 + 0.5 文案，长期内容产能瓶颈在文案（事件 + 结局段子）。

## 1.8 风险与对策

| 风险 | 等级 | 对策 |
|---|---|---|
| 内容消耗过快（49 事件不够撑住多周目体验） | 高 | 建立每周新增 5–8 条事件的内容节奏；事件去重池机制已实现 |
| 题材敏感（自嘲滑坡为对立煽动） | 中 | 所有结局/选项文案过价值观红线 review；保持"自嘲不嘲他人"原则 |
| 黑色幽默调性的受众有限 | 中 | 投放阶段精准定位即刻 / 小红书 / 微博职场圈层 |
| 技能 passive 用正则匹配文案，存在误判风险 | 低 | 计划在 v0.9 把识别迁移到事件元数据（tags 字段）而非文本正则 |
| 移动端 UI 适配（当前桌面优先） | 中 | 视口已锁，下阶段做 375px 宽度的专项 QA |

# 第二部分 · 玩法介绍（玩家视角）

## 2.1 一句话玩法

**选角色 → 选职业 → 一天 3 个时间槽，每槽抽一个事件，2–4 选项做出决策 → 数值变化 → 触发结局或撑过 14 天通关。死后业力可解锁技能和新职业，下一世更强。**

## 2.2 完整流程

```
[主菜单]
   │
   ├── 投胎 → 选角色（小马 / 小牛）
   │           └── 选职业（按业力世数解锁）
   │                  └── [游戏主页面]
   │                         │
   │                         ├── 每个时间槽：
   │                         │     ① 系统抽取一个未触发过的事件
   │                         │     ② 玩家点击 2–4 个选项之一
   │                         │     ③ 弹出 result 文本（如有）
   │                         │     ④ 触发 Toast / 扣款弹窗 / 绩效弹窗
   │                         │     ⑤ 推进到下一槽 / 下一天
   │                         │
   │                         ├── 每日开始：自动结算日薪
   │                         ├── Day 7：绩效弹窗（奖金 or 倒挂通知）
   │                         ├── 任意点：触发结局立即终结
   │                         └── Day 14 全程未死 → "活下来了"通关
   │
   ├── 嘴硬秘籍 → 技能树（消耗业力解锁）
   │
   └── 已故牛马名册 → 4 个标签页（怼史/排行/终局/事件）
```

## 2.3 一局示例

> **第 3 世**：小马 · IT 后端 · 已解锁"金钟罩"
>
> Day 1 上午 09:00 · 周一晨会
> 选项：「举手："去年的「再次出发」到底出发去哪了？"」（怼）
> → 心情 +14、压力 −3、工资分 −10
> → 因为是小马 + snark：心情再 +2、压力再 −2
> → 怼老板罚款：月薪 ¥12000 × 2.5% ≈ ¥300
> → 弹窗：💸 财务通知 −¥300，理由"老板觉得你「嗯」的语气不对"
>
> Day 1 下午 14:00 · 这个功能应该很简单吧
> 选项：「【橡皮鸭】当场把需求拆成 7 个子任务表」（技能选项）
> → 因为已解锁 rubber_duck，紫色选项可见
> → 技能 +8（passive 再 +2）、工资分 +6
>
> Day 7 早上：绩效弹窗"💰 +¥3000"
> ...
>
> Day 10 晚上：疲劳 100 + 健康 20 → 触发"过劳猝死"
> → 结局结算：本世业力 +13（10 天 + 新解锁 1 个结局 +5 − 已解锁多个）

## 2.4 多周目体验

| 解锁项 | 解锁条件 | 玩家增益 |
|---|---|---|
| 大厂运营 / IT 后端 / SaaS 销售 | 第 2 世起 | 月薪更高，事件池更丰富 |
| 乙方设计 | 第 3 世起 | 健康/心情起点最佳 |

## 3.2 数值系统

### 3.2.1 六大属性（0–100，全部 clamp）

| 属性 | 初值（默认） | 主要增长源 | 主要消耗源 | 阈值含义 |
|---|---|---|---|---|
| 健康 health | 80 | 副业反思、看病恢复 +5 | 加班、咖啡、通勤、看病前 | ≤25 + 疲劳 ≥100 → 过劳猝死 |
| 压力 stress | 30 | 老板、PM、加班 | 怼回去、摸鱼 | ≥100 + 心情 ≤20 → 精神崩溃 |
| 心情 mood | 70 | 怼回去、摸鱼、副业 | 被骂、被甩锅 | ≤10 + day≥5 → 抑郁结局 |
| 疲劳 fatigue | 20 | 加班、小组长每日+4 | 摸鱼、咸鱼心经 | ≥100 → 致死条件之一 |
| 技能 skill | 20 | 加班、橡皮鸭 passive | 长期不用 | ≤15 + day≥7 → 时代弃民 |
| 工资分 salary | 30 | 加班、绩效、画饼 | 怼老板、躺平 | ≤10 → 被优化；< initial → Day 7 倒挂 |

**初值来源**：取自所选职业的 `baseStats`，再叠加角色的 `statOffset`（小马 stress+5/mood+5，小牛 salary+5/fatigue+5）。

### 3.2.2 外部数值

| 字段 | 说明 |
|---|---|
| money | 存款，可负数；≤−300 触发"破产滚回老家" |
| salaryAmount | 月薪具体数字（来自 jobs.js），决定日薪和怼老板罚款 |
| dailyWage | `Math.round(salaryAmount / 22)`，每天自动入账 |
| medicalCost / snarkFine / healthDebt | 累计扣款明细，仅用于统计展示 |
| initialSalary | 投胎时工资分快照，Day 7 倒挂判定的对比基准 |
| inheritedMoney | 解锁"大圣归来"后继承上世存款 5%（仅 archive.lives[0]） |
| karma | 业力，跨世累积，购买技能消耗 |

## 3.3 角色系统

| 字段 | 小马 horse | 小牛 ox |
|---|---|---|
| Tagline | 烈但被世俗搓磨，暴怒只敢干吼 | 老好人，烂货都甩给你 |
| statOffset | stress +5, mood +5 | salary +5, fatigue +5 |
| snarkBonus（选择 snark 时） | mood +2, stress −2 | — |
| snarkPenalty（选择 snark 时） | — | salary −3, stress +3 |
| submissiveBonus（选择"忍"型文本时） | — | fatigue −3, salary +1 |
| submissivePenalty（选择"忍"型文本时） | mood −3 | — |

## 3.5 事件系统

### 3.5.1 事件数据结构

```js
{
  id: 'monday_morning_meeting',
  title: '周一晨会',
  text: '九点的会议室，空调还没暖起来。...',  // 情境描述
  timeSlot: 0,            // 限定时间槽 0/1/2（不填 = 全天可触发）
  minDay / maxDay,        // 限定出现日期范围
  once: true,             // 一次性事件
  pool: 'side_hustle',    // 事件池标记
  jobs: ['hr'],           // 仅特定职业触发
  client: true,           // 甲方挫折标记 → 负向 effect × 1.2
  choices: [...]
}
```

### 3.5.2 选项数据结构

```js
{
  text: '举手："去年的「再次出发」到底出发去哪了？"',
  snark: true,                    // 是否算"怼"，影响 history 和角色 trait
  hidden: true,                   // 是否为技能解锁的隐藏选项
  requiredSkill: 'thick_skin',    // 解锁该选项的技能 id
  effects: { mood: +14, stress: -3, salary: -10, money: -100 },
  result: '老板停顿了五秒，说"问得好"，然后接着讲他的远方。',
  moneyNote: '猎头费'              // 自定义 toast 备注
}
```

### 3.5.3 事件抽取逻辑（pickEvent）

```
1. 过滤所有不匹配当前 timeSlot / day / job / 技能门槛 的事件
2. 优先抽取 seenEventIds 中没有的事件
3. 如果全部见过，则在过滤后的池中随机
4. 池为空时返回 null → 自动 advanceTime 跳过
```

### 3.5.4 事件内容现状

- 主事件 25+，副业事件 5（side_didi / side_xhs / side_xianyu / side_writing / side_overload）
- 甲方专项事件 6（client_* 系列），均带 `client: true`
- 职业专项：team_lead_garbage、hr_self_layoff、scapegoat、rival_promotion 等
- 总事件数：**49 条**

## 3.6 效果计算管线（applyEffects）

按下列**顺序**叠加，每一步可能修改临时的 `adj` 对象，最后写入 state：

```
1️⃣  克隆 effects 到 adj
        ↓
2️⃣  甲方挫折放大
    若 event.client === true：负向 health/mood/fatigue/stress × 1.2
        ↓
3️⃣  角色性格修正
    若 choice.snark：叠加 character.traits.snarkBonus / snarkPenalty
    否则若文本匹配 submissive 正则：叠加 submissiveBonus / submissivePenalty
        ↓
4️⃣  技能 passive 修正
    coffee_immune  : 文本含咖啡 → health 不扣，fatigue 回复 × 1.5
    thick_skin    : snark 时 salary 损失 / 2
    rubber_duck   : 事件标题/文本含技术词 → skill +2
    social_butterfly : 含同事/团建/红包词 → mood +3
    promotion_radar : 含老板/画饼/HR 词 → salary +2
    iron_will     : health<30 时压力增量 / 2
        ↓
5️⃣  写入 state.stats（clamp 0~100），money 直接累加并触发 toast
        ↓
6️⃣  怼老板罚款
    snark === true → 扣 月薪 × 2.5%（四舍五入到百元），写入 lastChoiceCosts
        ↓
7️⃣  健康看病结算
    本次选项 health 净下降 drop > 0 → healthDebt += drop
    healthDebt 每攒满 10 触发一档：
      档1: ¥200    档2: ¥500    档3+: ¥1000
    扣款后 health 自动回复 +5，写入 lastChoiceCosts
        ↓
8️⃣  咸鱼心经 passive
    fishing_zen 已解锁 → 若 mood < 40，强制拉回 40
```

**前后端职责切分**：扣款明细统一写入 `state.lastChoiceCosts`，由 UI 层在 makeChoice 后弹出 deduction-modal。

## 3.7 结局系统

### 3.7.1 判定逻辑

每次 advanceTime 后，按 `priority` 倒序遍历 ENDINGS，**第一个命中的就是结局**，立即调用 finalizeLife 终止。

### 3.7.2 结局清单（按 priority）

| Priority | id | 名称 | 触发条件 |
|---|---|---|---|
| 100 | overwork_death | 过劳猝死 | fatigue≥100 && health≤25 |
| 95 | horse_slam_quit | 拍桌裸辞（小马专属） | character=horse && snarkCount≥5 && stress≥70 |
| 95 | ox_loyal_collapsed | 老实人之死（小牛专属） | character=ox && fatigue≥95 && salary≥60 |
| 90 | mental_collapse | 精神崩溃裸辞 | stress≥100 && mood≤20 |
| 88 | hr_self_optimized | 被自己优化（HR 专属） | job=hr && day≥5 && salary≤25 |
| 85 | fired_with_honor | 被开除·但走得有尊严 | salary≤15 && mood≥70 |
| 80 | optimized | 被优化 | salary≤10 |
| 75 | side_hustle_win | 副业反超·辞职自雇 | day≥10 && money≥5000 |
| 70 | era_abandoned | 不符合时代精神 | skill≤15 && day≥7 |
| 60 | broke | 破产滚回老家 | money≤−300 |
| 50 | depression | 陷入抑郁 | mood≤10 && day≥5 |
| 10 | survival | 活下来了（通关） | day>14 |

### 3.7.3 设计意图

- **高优先级抢断低优先级**：例如同时满足 overwork_death（100）和 broke（60），优先判过劳；
- **职业 / 角色专属结局**优先级介于通用致死和通用结局之间，保证差异化体验不被通用结局吞掉；
- **通关结局 priority=10**：兜底，只有所有死亡条件都没触发才会落到这里。

## 3.9 技能树系统

### 3.9.1 解锁规则

- **前置 OR 关系**：`canUnlockSkill` 中 `requires.some(unlocked.has)`，即只需前置中**任一项**已解锁；
- **业力消耗**：T1=10 / T2=20 / T3=30 / T4=50；
- **永久效果**：技能解锁后跨世生效，无消耗。

### 3.9.3 active 选项触达机制

`isChoiceVisible(choice)`：选项 `hidden=true` 时，仅当 `archive.unlockedSkills` 中包含 `choice.requiredSkill` 时显示。前端样式上加 `.skill-choice`（紫色）。

## 3.10 业力（Karma）系统

**累积来源**（每次 finalizeLife 时结算）：

| 来源 | 数值 |
|---|---|
| 存活基础 | day 数 × 1 |
| 首次解锁结局 | +5 |
| 首次解锁人设标签 | +2 |
| 首次触发事件 | +1 |

业力作为唯一的硬通货，用于技能树解锁。**没有"洗点"或"重置"机制**，所有消耗永久生效。

## 3.11 日薪 & Day 7 绩效系统

### 3.11.1 日薪

每次跨日（advanceTime 中 day 自增时）自动 `money += round(salaryAmount/22)`，并触发顶部 Toast。

### 3.11.2 Day 7 绩效结算

```js
if (day === 7 && !bonusPaidDay7) {
  if (currentSalary < initialSalary) {
    // 倒挂：不发钱，弹"绩效倒挂"通知
  } else {
    bonus = round(salaryAmount * currentSalary / 100 / 4);
    money += bonus;
    // 弹"绩效到账"通知
  }
}
```

- **倒挂"高情商"文案池**：5 条段子（如"做得越多越亏"）
- **正常奖金文案池**：5 条段子
- 弹窗机制：`pendingBonusModal` 入队，下次 renderGame 时优先弹出

## 3.12 罚款 & 看病系统（核心扣款）

### 3.12.1 怼老板罚款

- 触发条件：`choice.snark === true`
- 扣款公式：`round(salaryAmount × 2.5% / 100) × 100`（四舍五入到百元）
- 例：月薪 ¥12000 → 扣 ¥300；月薪 ¥3000 → 扣 ¥100
- 文案池：10 条戏谑理由（"PM 群里 at 你两次没收到"等）

### 3.12.2 健康看病

- 累计机制：所有 health 净下降进入 healthDebt
- 每攒满 10 触发一档扣款（不累进）：1×=¥200，2×=¥500，3×+=¥1000
- 扣款后**回血 +5**（让玩家感到"看病有效"）
- 文案池：1×9 条 / 2×5 条 / 3×3 条，按档位随机

### 3.12.3 扣款弹窗

- 一次选项触发多项扣款时，统一弹 deduction-modal 展示明细 + 合计 + 配套段子
- 段子池按扣款组合切换：纯怼 / 纯医疗 / 怼+医疗双重打击
- 玩家必须点击"认了"才继续推进

## 3.13 Toast 系统

- **队列模式**：避免连续 toast 互相覆盖；单条默认 1800ms
- **金钱变化自动归类**：正数 → 鼓励文案池 35 条（"钱包微笑/喜出望外"）；负数 → 沮丧文案池 35 条（"钱包流泪/月光预演"）
- **附带事件标题**：toast 文本含括号注明 `(工资 / 看病 / 副业...)`，让玩家知道钱花在哪
- **触达节点**：日薪入账、事件 effects 中的 money、副业收入、Day 7 奖金（弹窗+toast 二选一）

## 3.14 存档系统

### 3.14.1 双存档结构

| key | 内容 | 生命周期 |
|---|---|---|
| niuma_game_save_v3 | 当前进行中的一世（state） | 死亡时 clearState 清空 |
| niuma_game_archive_v2 | 跨世档案（archive） | 永久；销毁前世按钮可清空 |

### 3.14.2 archive 结构

```js
{
  totalLives,           // 累计死过几次
  lives: [{...}],       // 倒序排列，最近一世在 [0]
  unlockedEndings,      // 已解锁结局 id 列表
  unlockedTags,         // 已解锁标签 id 列表
  seenEventIds,         // 累计触发过的事件 id 列表
  karma,                // 业力余额
  unlockedSkills        // 已解锁技能 id 列表
}
```

### 3.14.3 容错

- 每个加载点都 try/catch，损坏时回退默认；
- 字段缺失用 `?? defaultValue`，平滑升级；
- 销毁前世 = 直接 `localStorage.removeItem` 两个 key。
